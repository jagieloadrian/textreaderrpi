## Objective

Evolve the existing DisplayDriver interface from a single scrollText() method to a hybrid contract supporting clear(), write(), and status() operations. Upgrade MAX7219 implementation to conform. This establishes the foundation for multi-driver support (LCD, OLED) in subsequent plans.

**Purpose:** Enable uniform control across multiple display types per D-01 (hybrid contract) + D-02 (mandatory methods).  
**Output:** Updated DisplayDriver.kt + upgraded Max7219Display.kt with full backward compatibility + test coverage.

## Context

**Current state:** Phase 1 complete. MAX7219 driver exists in `src/main/kotlin/com/anjo/driver/` using scrollText() exclusively.  
**Upgrade path:** Add clear(), write(text), status() to interface + implement in MAX7219.

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Server | HTTP GET /api/text, POST /api/text (user-submitted text) |
| Server → Hardware | GPIO/SPI to I2C devices (I2C control protocol) |

### STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | GPIO/SPI write buffer | mitigate | Max7219Display caches GPIO state; validate buffer before send; no direct external control of GPIO |
| T-02-02 | DoS | Hardware unavailability on boot | mitigate | Per D-09: fail-fast (don't block server init); log error; gracefully degrade to offline mode |
| T-02-03 | Information Disclosure | Status endpoint leaks hardware details | accept | status() returns basic state only (no GPIO values, no raw I2C data); low-value information for attacker |
