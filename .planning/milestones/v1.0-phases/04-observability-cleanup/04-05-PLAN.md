<objective>
Refactor RecoveryPolicy for readability and clearer control flow (D4-15, D4-26), refactor ResourceTracker toward monitoring-oriented usage with metrics exposure (D4-16, D4-27), and simplify recovery policy usage in ScreenDriverService (D4-25). ResourceTracker is NOT removed — it remains pending user confirmation (D4-28, G4-01).
</objective>

<context>
RecoveryPolicy is a 66-line class with exponential backoff retry logic. It's already clean but has some areas for improvement: magic numbers, inline jitter calculation, and verbose logging patterns. ResourceTracker is a 54-line bounded slot tracker. It needs to expose observable state for the metrics endpoint (created in 04-04) and register gauges. ScreenDriverService manually wraps `recoveryPolicy.execute(...)` inline — D4-25 suggests moving toward annotation-driven or cleaner integration.
</context>
