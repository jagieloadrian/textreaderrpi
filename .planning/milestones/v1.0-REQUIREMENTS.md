## Validated Acceptance Criteria

### MVP Acceptance
- [x] User can submit text via HTTP POST /api/text
- [x] Text appears on MAX7219 LED matrix within 2 seconds
- [x] Scrolling animation is smooth and readable
- [x] Server recovers gracefully from network interruptions
- [x] No memory leaks after 24 hours of operation
- [x] Unit tests for critical services (>70% coverage)

### Enhanced Display Support Acceptance
- [x] I2C 16x2 LCD renders text correctly
- [x] OLED display shows text with correct formatting
- [x] Configuration file controls display type selection
- [x] Switching display types doesn't require code changes
- [x] All display drivers share common interface
- [x] `GET /`, `GET /status`, and `GET /settings/display` render responsive HTML pages
- [x] HTML pages are generated with Ktor HTML DSL templates using shared layout + route-specific sections
- [x] Browser-oriented 400/500 responses render HTML error pages

### Production Readiness Acceptance
- [x] `/health` endpoint reports system status
- [x] Error logging captures hardware failures
- [x] Rate limiting prevents API abuse
- [x] Deployment instructions documented
- [x] README includes hardware setup guide

### Phase 4-5 Acceptance (added during development)
- [x] `/metrics` returns runtime + API + hardware metric groups in JSON
- [x] Scheduling engine supports ONESHOT, RECURRING, and CRON types
- [x] Effects are selectable via POST body (`effect` field)
- [x] All config externalized to env vars
- [x] Docker image builds via Gradle

## Requirements That Evolved

- **§2.8 Scheduled text updates** (originally Phase 3 Feature) — grew into a full Phase 5 deliverable with its own service, repository, conflict policy, and UI
- **§2.8 Multiple concurrent displays** — deferred to future milestone (not in scope for v1.0)
- **Software Constraints** (REQUIREMENTS.md §4) — Kotlin 2.1.0/Ktor 2.3.12 updated to Kotlin 2.3.21/Ktor 3.5.0/Pi4J 4.0.0 during Phase 5
