# Plan 05-05: SchedulerService Coroutine Engine + Conflict Policy

## Objective

Implement `SchedulerService`: the coroutine-based job engine that fires schedules from the database (D5-12).
Implement conflict policy: ad-hoc POST /api/text preempts ongoing scheduled display; interrupted schedules re-queue (D5-15).
Implement priority ordering: higher priority wins, earlier creation wins on tie (D5-17).
Support one-shot, recurring (interval strings), and cron triggers (D5-09).
Recurring schedules run indefinitely unless `expiresAt` set or deleted (D5-18).

## Background

This is the core scheduling engine. It scans the database for due schedules, fires them in priority order, and manages the conflict policy when ad-hoc text requests arrive. The scheduler lifecycle is tied to the Ktor application (start on ApplicationStarted, cancel all on ApplicationStopping).

  <action>
    Create `src/main/kotlin/com/anjo/service/SchedulerService.kt`.
    Package: `package com.anjo.service`
    Imports: kotlinx.coroutines (CoroutineScope, Dispatchers, SupervisorJob, launch, delay, isActive, Job, CancellationException), com.anjo.db.ScheduleRepository, com.anjo.model.{Schedule, TriggerType, ScheduleStatus}, java.time.{Instant, Duration}, com.cronutils (CronParser, CronDefinitionBuilder, CronType, ExecutionTime), java.time.ZonedDateTime, java.util.concurrent.ConcurrentHashMap

    Define `class SchedulerService(private val repository: ScheduleRepository, private val screenService: ScreenDriverService)`:

    Fields:
    - `private val scope = CoroutineScope(Dispatchers.Default + SupervisorJob())`
    - `private val activeJobs = ConcurrentHashMap<String, Job>()`
    - `private val cronParser = CronParser(CronDefinitionBuilder.instanceDefinitionFor(CronType.UNIX))`

    Public functions:
    - `fun start()` — calls `scope.launch { tickLoop() }`; then loads and schedules all ACTIVE schedules from DB
    - `fun stop()` — calls `scope.cancel()`; clears `activeJobs`
    - `fun schedule(schedule: Schedule)` — launches the correct coroutine type (oneshot/recurring/cron) and stores the Job in activeJobs
    - `fun cancel(id: String)` — cancels and removes the job for the given schedule id; calls `repository.updateStatus(id, "DONE")`
    - `suspend fun requeueAfterInterruption(id: String)` — calls repository to set status back to ACTIVE; re-schedule the job

    Private functions:
    - `private suspend fun tickLoop()` — runs every minute; checks all active schedules for missed fires (handles edge case of app restart)
    - `private fun launchOneShot(schedule: Schedule)` — `scope.launch { delay(targetMs - now); fire(schedule) }`; `targetMs` parsed from `schedule.triggerValue` as ISO8601 Instant
    - `private fun launchRecurring(schedule: Schedule)`:
      - Parse interval: "5m" → 5 minutes, "1h" → 1 hour, "30s" → 30 seconds (simple suffix parsing)
      - `scope.launch { var runs = 0; while (isActive) { delay(intervalMs); if (expiresAt != null && Instant.now().isAfter(Instant.parse(expiresAt))) break; if (maxRuns != null && runs >= maxRuns) break; fire(schedule); runs++ } }`
    - `private fun launchCron(schedule: Schedule)`:
      - Parse cron expression via `cronParser.parse(schedule.triggerValue)`
      - `scope.launch { while (isActive) { val next = ExecutionTime.forCron(cron).nextExecution(ZonedDateTime.now()).orElseThrow(); val delayMs = next.toInstant().toEpochMilli() - System.currentTimeMillis(); if (delayMs > 0) delay(delayMs); fire(schedule) } }`
    - `private suspend fun fire(schedule: Schedule)`:
      - Call `screenService.displayScheduled(schedule.text, schedule.id, schedule.effect)`
      - Update DB status to DONE for one-shot after firing

    Add a `suspend fun displayScheduled(text: String, scheduleId: String, effect: String)` method to `ScreenDriverService` that:
    - Sets `currentScheduledId = scheduleId`, registers `currentDisplayJob`
    - Acquires `displayMutex.withLock { callExistingDisplayMethod(text) }`
    - Returns normally (no return value needed)
    - If cancelled via CancellationException, does NOT re-throw (the scheduler handles re-queue)
  </action>
