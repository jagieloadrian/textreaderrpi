## Objective
Extend /api endpoints to support display management: GET /api/display/status (query state), POST /api/display/select (switch drivers). Implement queue-based runtime switching per D-04.
## Tasks
### Task 1: Create DisplayApi request/response types + endpoints
- Create DisplayApi.kt with data classes: DisplayStatusResponse, DisplaySelectRequest, DisplaySelectResponse
- Add GET /api/display/status endpoint (returns JSON)
- Add POST /api/display/select endpoint (validates driver, queues switch)
- Whitelist allowed drivers (max7219, lcd, oled)
- Return 400 on invalid driver type
### Task 2: Implement queue + switch logic
- Add queueDisplaySwitch(type) to ScreenDriverService
- Queue switch if driver busy; switch immediately if idle (per D-04)
- Update scrollText()/write() to call checkAndPerformPendingSwitch() after completing
- Single-slot queue (no unbounded queue)
- Mock & test all scenarios
## Success Criteria
✓ GET /api/display/status returns DisplayStatusResponse JSON
✓ POST /api/display/select accepts driver, returns success/error
✓ Invalid driver returns 400 + error
✓ queueDisplaySwitch() works correctly (queues when busy, switches when ready)
✓ Tests pass with >80% coverage
✓ /api/text unchanged (backwards compatible)
✓ Build succeeds
